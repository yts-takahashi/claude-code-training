# ハンズオン02: 画像/モックアップからClaude CodeでUI生成

## 目的

画面デザインのモックアップ（スクリーンショット）をClaude Codeに渡し、本格的なReactコンポーネントを生成する体験をします。

このハンズオンを通じて、以下を習得します:

- 画像からUIコードを生成するワークフロー
- デザイン再現度を高めるプロンプトの書き方
- 生成されたコードのコンポーネント分割指示
- 複数画面にわたる共通コンポーネントの再利用

## 背景: MUFGの事例

三菱UFJフィナンシャル・グループ（MUFG）では、FigmaのデザインデータからPNG画像をエクスポートし、Claude（当時はClaude 3.5 Sonnet）に渡すことでReactコンポーネントを生成するワークフローを実践しています。

この手法のポイント:
- **Figmaのデザインデータを直接渡すのではなく、PNGスクリーンショットを使う** -- LLMは画像を理解できるため、デザインツールのネイティブ形式は不要
- **1画面ずつ段階的に生成する** -- 最初の画面で共通コンポーネント（サイドバー、ヘッダー等）が生成され、後続の画面でそれらが再利用される
- **生成後に人間がレビュー・調整する** -- 100%の再現は目指さず、80-90%の精度で高速にプロトタイピング

このハンズオンでは、同様のワークフローを金融ダッシュボードの6画面で実践します。

## 事前準備

### 1. モックアップのスクリーンショットを撮る

`mockups/` ディレクトリに6つのHTMLファイルがあります。それぞれをブラウザで開き、スクリーンショットを撮ります。

```bash
# macOS の場合、ブラウザで開く
open mockups/01-dashboard.html
```

**スクリーンショットの撮り方:**

- **macOS**: `Cmd + Shift + 3`（全画面）または `Cmd + Shift + 4`（範囲選択）
- **Chrome DevTools**: `Cmd + Shift + I` → `Cmd + Shift + P` → "Capture full size screenshot"
- ブラウザのウィンドウサイズを **1920x1080** に合わせると、最も再現しやすい結果が得られます

> **ヒント**: Chrome DevToolsの「Capture full size screenshot」を使うと、スクロール領域も含めた完全なスクリーンショットが撮れます。

### 2. プロジェクトのセットアップ

```bash
cd project/
npm install
npm run dev
```

開発サーバーが `http://localhost:3000` で起動することを確認してください。

### 3. Claude Codeの準備

Claude Codeを `project/` ディレクトリで起動します。

```bash
cd project/
claude
```

`CLAUDE.md` が自動的に読み込まれ、技術スタックやコーディング規約がClaude Codeに伝わります。

## Step 1: ダッシュボード画面の生成（30分）

**なぜ最も複雑な画面から始めるのか:**

ダッシュボード画面には、サイドバー、ヘッダー、KPIカード、チャート、テーブルなど、アプリ全体で使う要素が凝縮されています。この画面を最初に生成することで:
- 共通レイアウト（サイドバー + ヘッダー）が定義される
- 基本的なUIコンポーネント（Card, Badge, Table）が生まれる
- デザインの基調（カラー、フォント、間隔）がコードに反映される

### プロンプト例

`01-dashboard.html` のスクリーンショットを撮影した上で、以下のようにClaude Codeに指示します:

```
この画像は金融ダッシュボードアプリのメイン画面のデザインモックアップです。
このデザインをNext.js + TypeScript + Tailwind CSSで実装してください。

実装要件:
- App Router を使用
- サイドバーとヘッダーは共通レイアウトとして app/layout.tsx に配置
- KPIカードは再利用可能なコンポーネントとして実装
- チャートは Recharts を使用（Chart.jsではなく）
- テーブルデータはダミーデータとして lib/data/ に分離
- デザインのカラー、間隔、フォントサイズを可能な限り正確に再現
- 日本語テキストはモックアップと同じ内容を使用
```

### チェックポイント

生成されたコードを以下の観点で検証してください:

- [ ] ブラウザで表示した際、モックアップと見比べてレイアウトが一致しているか
- [ ] サイドバーのナビゲーション項目が全て揃っているか
- [ ] KPIカードの値・アイコン・色が正しいか
- [ ] チャートが表示されているか（データの正確さより描画されることが重要）
- [ ] テーブルのカラム構成とステータスバッジが再現されているか
- [ ] レスポンシブ対応がある程度効いているか

## Step 2: コンポーネント分割を指示（15分）

Step 1で生成されたコードは、1ファイルに多くの要素が含まれている可能性があります。次のステップでは、コンポーネント分割を指示します。

### プロンプト例

```
生成されたダッシュボード画面のコードをコンポーネントに分割してください。

分割方針:
- components/layout/Sidebar.tsx - サイドバーナビゲーション
- components/layout/Header.tsx - ヘッダー（検索バー、通知、ユーザー情報）
- components/ui/KpiCard.tsx - KPI表示カード（汎用的に使えるように）
- components/ui/StatusBadge.tsx - ステータスバッジ（約定済、処理中、取消済など）
- components/ui/DataTable.tsx - データテーブル（ソート・ページネーション対応の骨組み）
- components/charts/AssetLineChart.tsx - 資産推移の折れ線チャート
- components/charts/SectorPieChart.tsx - セクター別配分の円グラフ

各コンポーネントは適切なPropsの型定義を持つこと。
```

### チェックポイント

- [ ] 各コンポーネントが独立したファイルに分離されているか
- [ ] Props の型定義が interface で定義されているか
- [ ] 分割後もブラウザ表示が崩れていないか
- [ ] import/export が正しく機能しているか

## デザイン再現度を上げるコツ

### 1. スクリーンショットの品質

- **解像度を高く**: Retinaディスプレイなら2倍解像度で撮影
- **全画面で撮る**: 部分的なスクリーンショットではなく、画面全体を渡す
- **フルスクリーンショット**: Chrome DevToolsの "Capture full size screenshot" でスクロール全体を含める

### 2. プロンプトの工夫

**具体的な指示を追加する:**
```
# 悪い例
この画像を再現してください

# 良い例
この画像は金融ダッシュボードのメイン画面です。
以下の点に特に注意して実装してください:
- KPIカードのアイコンの背景色は、各カードで異なる色を使用
- テーブルの偶数行には薄いグレーの背景を適用
- ステータスバッジは角丸のpillスタイル
- 金額は3桁区切りのカンマフォーマット
```

**フォント・カラーを明示する:**
```
デザインシステム:
- フォント: Noto Sans JP
- プライマリカラー: navy-900 (#102a43)
- アクセントカラー: blue-500 (#3b82f6)
- 成功: emerald-600, 警告: amber-600, エラー: red-600
- 角丸: xl (12px) をカードに、lg (8px) をボタンに使用
```

### 3. 反復的な改善

一度の生成で完璧なUIは得られません。以下のサイクルで改善します:

1. **生成** -- スクリーンショットを渡して初回生成
2. **比較** -- モックアップと生成結果をブラウザで並べて比較
3. **差分を指示** -- 「KPIカードの間隔がモックアップより広い。gap-6をgap-4に変更して」
4. **再生成** -- 修正を適用して再確認

### 4. デバッグのコツ

```
# レイアウトの問題を特定するプロンプト
現在の画面をモックアップと比較すると、サイドバーの幅が狭く見えます。
モックアップではサイドバーが w-64 (256px) ですが、
現在の実装を確認して修正してください。

# チャートの表示問題
Rechartsの折れ線チャートが表示されません。
コンソールエラーを確認し、修正してください。
データは12ヶ月分のダミーデータを使用してください。
```

## まとめ

このハンズオンで体験した「画像 → コード生成」のワークフローは、実際の開発現場で以下のような場面で活用できます:

- **プロトタイピング**: デザイナーが作成したFigmaデザインから、数時間でインタラクティブなプロトタイプを構築
- **デザインシステムの初期構築**: 既存のデザインモックアップから、コンポーネントライブラリの雛形を生成
- **レガシーUI刷新**: 現行システムのスクリーンショットから、モダンなフレームワークへの移行コードを生成

重要なのは、**AI生成はゴールではなくスタート地点** だということです。生成されたコードを土台として、人間が品質・アクセシビリティ・パフォーマンスを磨き上げていくプロセスが本質です。
