なぜコンポーネント再利用が重要か

自前実装の問題

  • デザインの一貫性が失われる
  • バグ修正が各所に分散
  • テスト・メンテナンスコスト増大
  • AIが毎回ゼロから生成してしまう

コンポーネント再利用の利点

  • UI / UX の統一性を担保
  • 修正が一箇所で済む
  • 開発スピードの向上
  • AIの生成精度が向上
課題: AIコード生成ツールは、指示しなければ既存コンポーネントを無視して自前実装しがち。
解決策: CLAUDE.md + 明示的なプロンプト で再利用率を最大化する。

CLAUDE.md の役割と重要性

CLAUDE.md は Claude Code にプロジェクトの規約・制約を伝える最も効果的な手段

CLAUDE.md に書くべき内容

  • コンポーネント一覧と用途
    セレクタ、パス、役割を明記
  • 必須ルール
    「既存コンポーネントを必ず使用すること」
  • import パス規約
    @/components/ComponentName 形式
  • スタイリング規約
    Tailwind CSS + fin-* カラー
  • ドメイン固有の表示規約
    金額→CurrencyDisplay、状態→StatusBadge

効果

CLAUDE.md なし

AIが自前で <table> や色分けを実装

再利用率: 低い

CLAUDE.md あり

AIが既存コンポーネントを優先的に使用

再利用率: 高い

事前準備

  1. 依存パッケージのインストール
    cd handson-03-storybook-components npm install
  2. Storybookの起動確認
    npm run storybook # ブラウザで http://localhost:6006 にアクセス
  3. プロジェクト構成を確認
    handson-03-storybook-components/ ├── CLAUDE.md # Claude Code用プロジェクト設定 ├── src/ │ ├── components/ # FinUIコンポーネントライブラリ (15個) │ └── main.ts ├── exercises/ # 演習課題 (3つ) ├── .storybook/ ├── package.json └── tailwind.config.js

Step 1: Storybookでコンポーネント確認

  • Storybookを起動し、各コンポーネントのストーリーを確認する
  • 各コンポーネントの @Input / @Output とバリアントを把握する

確認すべきポイント

  • デフォルトのストーリー
  • 全バリアントの一覧
  • インタラクション例
  • 金融ドメインに即したサンプルデータ

Storybook MCP（オプション）

  • @Input/@Output定義を自動的に把握
  • ストーリーから使い方パターンを学習
  • バリアント（variant, sizeなど）の正確な指定

※ MCP なしでも CLAUDE.md で代替可能

Tip: Storybook の Controls パネルで各 @Input のデフォルト値・型・選択肢を確認しておくと、プロンプトの精度が上がります。

Step 2: CLAUDE.md の内容確認

必須ルール

  • 新規画面では必ず既存コンポーネントを使用
  • 同等機能の自前実装は禁止

import 規約

// 個別インポート（推奨） import { ButtonComponent } from '@/components/Button'; // standalone imports imports: [ButtonComponent, ...]

表示規約

  • 金額 → fin-currency-display
  • ステータス → fin-status-badge
  • 承認フロー → fin-timeline

スタイリング規約

  • Tailwind CSS のユーティリティクラスを使用する
  • カスタムカラーは fin-* プレフィックス（例: text-fin-primary, bg-fin-danger）
  • レイアウトには Tailwind の Flex / Grid を使用する
  • インラインスタイルは原則禁止

Exercise 1: 融資申請一覧画面

使用コンポーネント

  • fin-data-table — 申請データの一覧表示
  • fin-status-badge — ステータス表示
  • fin-search-input — 検索（デバウンス300ms）
  • fin-button — アクションボタン
  • fin-currency-display — 融資金額の表示

機能要件

  • 検索（申請番号 / 申請者名、部分一致）
  • ステータスフィルター（6種別）
  • カラムソート（昇順 / 降順 / 初期状態）
  • ページネーション（10件 / ページ）
  • 複数行選択 + 一括承認

プロンプト例

src/pages/loan-application-list .component.ts を作成してください。 exercises/exercise-01.md の要件に従い、 融資申請一覧画面を実装してください。 必ず既存の FinUI コンポーネント （DataTableComponent, StatusBadgeComponent, SearchInputComponent, ButtonComponent, CurrencyDisplayComponent） を使用してください。 サンプルデータは15件以上用意し、 検索・フィルター・ソート・ ページネーションの機能を 実装してください。

Exercise 1: 検証チェックリスト

コンポーネント使用チェック

  • 既存コンポーネント5つがすべて imports に含まれているか
  • <table> タグの直接使用がないか
  • ステータスの色分けを自前で実装していないか
  • import パスが @/components/xxx 形式か

@Input の使用チェック

  • fin-data-table の [data], [columns], [pagination], [selectable]
  • fin-status-badge の [status]
  • fin-search-input の [debounceMs]
  • fin-button の [variant], [size]

機能チェック

  • 検索がデバウンスで動作するか
  • ステータスフィルターが正しく絞り込みできるか
  • ソートが各カラムで動作するか
  • ページネーションが正しく動作するか
  • 行選択と一括操作が連動しているか
  • 金額が fin-currency-display で表示されているか
よくある問題: AIが <table> を直接生成してしまう場合は、プロンプトで「DataTableComponent を使うこと」と再指示

コンポーネント再利用率を上げる 5つのコツ

  1. CLAUDE.md に明確なルールを書く — 「必ず既存コンポーネントを使用すること」＋コンポーネント一覧表
  2. プロンプトでコンポーネント名を明示する
    # 効果が低い 「融資一覧画面を作ってください」 # 効果が高い 「DataTableComponent, StatusBadgeComponent, SearchInputComponent を使って融資一覧画面を作ってください」
  3. 要件にコンポーネントのマッピングを含める — 「融資金額カラムは fin-currency-display で表示」と明記
  4. Storybook のストーリーを参照させる — 「data-table.stories.ts のサンプルデータ構造を参考に」
  5. 段階的に構築する — ヘッダー → KPIカード → チャート → テーブルと分割して生成

AIが既存コンポーネントを無視するとき

症状1: 自前で <table> を生成

DataTableComponent を使わず HTML テーブルを直接記述

対処:

「fin-data-table コンポーネントを使って書き直してください。自前のテーブルは使わないでください」

症状2: ステータスの色分けを自前実装

ngClass で色を切り替える独自実装

対処:

「ステータス表示は fin-status-badge を使い、[status] に値を渡してください」

症状3: import パスが異なる

相対パスや存在しないパスからインポート

対処:

CLAUDE.md の import 規約を再確認させる。@/components/xxx 形式を明示

症状4: コンポーネントの @Input を誤用

存在しない @Input を指定する

対処:

「Storybook の stories ファイルを確認し、正しい @Input を使ってください」

根本対策: CLAUDE.md をできるだけ具体的に記述し、コンポーネントの @Input 一覧とサンプルコードを含める
