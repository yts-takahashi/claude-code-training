# ハンズオン01: Excel仕様書からClaude Codeでアプリを生成する

## 目的

エンジニアマネージャーが、実際の業務で使われるExcel仕様書をClaude Codeに読み込ませ、Webアプリケーションのコードを生成する一連の流れを体験します。

このハンズオンを通じて以下を学びます:

- Claude Code がExcelファイルを読み取り、構造化されたデータを理解する能力
- 仕様書からDBスキーマ・API・画面を段階的に生成する進め方
- AIが生成したコードの検証方法とハルシネーション（幻覚）の検出方法
- プロンプトの書き方による出力品質の違い

## 前提条件

### 環境

- Claude Code がインストール済みであること
- Node.js 18以上がインストール済みであること
- Angular CLI がインストール済みであること（`npm install -g @angular/cli`）
- NestJS CLI がインストール済みであること（`npm install -g @nestjs/cli`）
- PostgreSQL が利用可能であること（ローカルまたはDocker）

### 事前準備

```bash
# このディレクトリに移動
cd handson-01-excel-spec

# （オプション）仕様書を再生成する場合
pip install openpyxl
python generate_spec.py
```

### ファイル構成

```
handson-01-excel-spec/
  CLAUDE.md                          # Claude Code 用プロジェクト設定
  README.md                          # このファイル（ハンズオン手順書）
  generate_spec.py                   # Excel仕様書生成スクリプト
  specs/
    融資管理システム仕様書.xlsx         # 生成済み仕様書（7シート）
```

## ハンズオン手順

### Step 0: Claude Code の起動とプロジェクト初期化

```bash
# このディレクトリで Claude Code を起動
cd handson-01-excel-spec
claude
```

Claude Code が起動したら、まず Angular + NestJS のモノレポプロジェクトを初期化させます:

```
Angular 17+ (standalone components) のフロントエンドと NestJS のバックエンドで構成される
モノレポプロジェクトを初期化してください。

- frontend/ ディレクトリに Angular CLI で新規プロジェクトを作成（Tailwind CSS を設定）
- backend/ ディレクトリに NestJS CLI で新規プロジェクトを作成（Prisma をセットアップ）
- パッケージマネージャーは npm を使ってください
```

> **ポイント**: CLAUDE.md が自動的に読み込まれ、プロジェクトの前提条件がAIに伝わります。

### Step 2: DB スキーマ生成を指示

#### プロンプト例

```
specs/融資管理システム仕様書.xlsx の「テーブル定義」シートを読み込んで、
Prisma スキーマファイル（backend/prisma/schema.prisma）を生成してください。

以下の要件を守ってください:
- テーブル定義シートの全テーブル・全カラムを漏れなく定義すること
- 外部キー制約はPrismaのリレーションとして定義すること
- インデックスは @@index で定義すること
- データ型はPostgreSQL互換のPrisma型にマッピングすること
- カラム名はスネークケースのまま @map は不要（DBスキーマなので）
- コメントを付与すること
```

#### 検証チェックポイント

- [ ] 全8テーブルが定義されているか
- [ ] 各テーブルのカラム数が仕様書と一致しているか（特に customers: 21カラム、loan_applications: 26カラム）
- [ ] 外部キーが正しくリレーションとして定義されているか
- [ ] インデックスが仕様書通りに定義されているか
- [ ] NULL可否がOptional（?）で正しく表現されているか
- [ ] デフォルト値が `@default()` で設定されているか
- [ ] **仕様書にないテーブルやカラムが追加されていないか**（ハルシネーション検出）

#### よくあるハルシネーション

- `users` テーブルや `branches` テーブルを勝手に作成する（外部キー参照先だが仕様書に定義がない）
- `audit_logs` テーブルを追加する（機能一覧に記載はあるがテーブル定義シートにはない）
- カラムの型が微妙に異なる（例: `VARCHAR(20)` を `String` にするのは正しいが、長さ制約を `@db.VarChar(20)` で付けるかどうか）

### Step 3: API 実装を指示

#### プロンプト例

```
specs/融資管理システム仕様書.xlsx の「API仕様」シートを読み込んで、
NestJS の Controller + Service として API を実装してください。

まず融資申請関連の API（No.3〜8）を実装してください:
- GET /api/loan-applications（一覧取得・ページネーション・フィルタ対応）
- GET /api/loan-applications/:id（詳細取得）
- POST /api/loan-applications（新規作成）
- PUT /api/loan-applications/:id（更新）
- DELETE /api/loan-applications/:id（削除）
- POST /api/loan-applications/:id/submit（申請提出）

以下の構成で実装してください:
- backend/src/modules/loan-applications/ 配下にモジュール・コントローラー・サービスを作成
- リクエストのバリデーションは class-validator の DTO クラスで定義
- Prisma でのデータベース操作はサービス層に集約
- NestJS の Exception Filter による適切なエラーハンドリング
- レスポンス形式は仕様書通り
- @UseGuards(JwtAuthGuard) による認証チェック
```

#### 検証チェックポイント

- [ ] APIパスが仕様書と完全に一致しているか
- [ ] HTTPメソッドが正しいか（`@Get()`, `@Post()`, `@Put()`, `@Delete()` デコレーター）
- [ ] リクエストパラメータが仕様書通りか（DTO に定義されているか）
- [ ] レスポンス形式が仕様書の定義と一致しているか
- [ ] ステータスコードが仕様書の全パターンをカバーしているか
- [ ] 認証チェックが含まれているか（`@UseGuards(JwtAuthGuard)`）
- [ ] 権限マトリクスに基づいた権限チェックが含まれているか（`@Roles()` デコレーター等）
- [ ] **仕様書にないクエリパラメータやレスポンスフィールドが追加されていないか**

#### よくあるハルシネーション

- 仕様書にないフィルタパラメータを追加する（例: `sort_by` や `search` が仕様書の記述と微妙に異なる）
- レスポンスに仕様書にないフィールドを追加する（例: `_count` や `_links`）
- DELETE APIで「下書きのみ削除可能」のビジネスルールが欠落する

- コンポーネントは standalone: true で作成すること

### Step 5: ビジネスロジック実装を指示

#### プロンプト例

```
specs/融資管理システム仕様書.xlsx の「ビジネスルール」シートを読み込んで、
以下のビジネスロジックを NestJS のサービスとして実装してください:

1. 与信スコア算出（BR-001〜BR-003, BR-016）
2. 融資限度額算出（BR-004）
3. 適用金利決定（BR-005）
4. 元利均等返済額計算（BR-006）
5. 承認フロー分岐（BR-007）

各ロジックは:
- backend/src/modules/ 配下の適切なモジュールのサービスに配置
  （例: backend/src/modules/credit/credit-score.service.ts）
- @Injectable() デコレーターを付与し、DI 可能にする
- 計算ロジックは純粋関数としてメソッド内に実装（テストしやすく）
- 計算式は仕様書の記載を正確に実装
- Jest によるユニットテストも同時に作成（*.spec.ts）
```

#### 検証チェックポイント

- [ ] 与信スコアの各サブスコアの閾値が仕様書と完全一致しているか
  - 例: 年収300万未満=50、300-500万=100 ...（BR-002）
- [ ] 融資限度額の倍率が正しいか（スコア800以上で年収x8 等）
- [ ] 金利計算の調整値が正しいか（スコア800以上で-0.3% 等）
- [ ] 元利均等返済の計算式が正しいか（`P * r * (1+r)^n / ((1+r)^n - 1)`）
- [ ] 承認フロー分岐の金額閾値が正しいか（500万 / 3000万）
- [ ] 例外条件が実装されているか（自営業の場合、公務員ボーナス等）
- [ ] **計算式が仕様書と異なる独自のロジックになっていないか**

#### よくあるハルシネーション

- 与信スコアの閾値を「一般的な値」に置き換える（仕様書の値ではなく）
- 返済計算で独自の丸め処理を追加する
- 承認フローの金額境界値を変更する（例: 500万を1000万にする）
- 仕様書にない追加ルール（例: 年齢による制限）を勝手に追加する

## ハルシネーション検出のコツ

Claude Code を含む AI コード生成ツールは、仕様書にない情報を「補完」する傾向があります。
以下のポイントに注意して、生成されたコードを検証してください。

### 1. 数値の正確性を最優先で確認する

ビジネスルールに含まれる数値（閾値・倍率・金利等）は、仕様書と1つずつ突き合わせてください。
AIは「それっぽい値」を生成することがあり、仕様書の値と微妙に異なる場合があります。

```
確認例:
仕様書: "年収300万未満:50 / 300-500万:100"
生成コード: if (income < 3000000) return 50;  // OK
            if (income < 5000000) return 100;  // NG: "300-500万" は 300万以上500万未満
```

> **万円単位と円単位の変換ミス** は特に頻出します。仕様書が「万円」で記載されている場合、コード内で正しく変換されているか必ず確認してください。

### 2. カラム数・API数を数える

仕様書に記載されたカラム数やAPIエンドポイント数を事前に数えておき、生成結果と比較します。
数が多い場合は「AIが勝手に追加した」可能性があり、少ない場合は「見落とし」の可能性があります。

### 3. 「仕様書にないもの」を探す

AIは「こうあるべき」と判断して、仕様書にない機能やフィールドを追加することがあります:
- 仕様書にない `updatedAt` や `createdBy` カラムの追加
- 仕様書にないAPIエンドポイントの追加（例: PATCH メソッド）
- 仕様書にない画面項目の追加（例: 検索候補のサジェスト機能）
- 仕様書にないバリデーションルールの追加

これ自体が悪いわけではありませんが、「仕様書に基づく実装」として適切かどうかを判断する必要があります。

### 4. 境界値に注目する

- 「以上」「以下」「未満」「超」の区別
- 金額の境界値（500万「以下」なのか「未満」なのか）
- 0件・NULL・空文字のハンドリング

### 5. 仕様書を参照させながらレビューする

生成されたコードについて、Claude Code 自身にレビューさせることも効果的です:

```
生成した与信スコア算出ロジック（backend/src/modules/credit/credit-score.service.ts）を、
仕様書の「ビジネスルール」シート BR-001〜BR-003 と突き合わせて、
差異がないか確認してください。差異がある場合は具体的に指摘してください。
```

### 6. テストケースで検証する

ビジネスルールの検証には、仕様書の計算式から期待値を手計算し、テストケースとして追加するのが最も確実です:

```
以下のテストケースを追加して、与信スコア算出が仕様書通りか検証してください:
- 年収400万・正社員・勤続5年・負債比率0.15・担保なし → 期待スコアを手計算して比較
```

## 補足: 効果的なプロンプトのパターン

### 悪い例

```
仕様書を読んでアプリを作って
```

問題点: 範囲が広すぎて、AIが独自の判断で多くの仕様を「補完」してしまう。

### 良い例

```
specs/融資管理システム仕様書.xlsx の「テーブル定義」シートにある
loan_applications テーブルの定義を読み込んで、
Prisma スキーマの model を1つだけ生成してください。
配置先は backend/prisma/schema.prisma です。
カラムは仕様書に記載されたものだけを含め、追加しないでください。
```

ポイント:
- **対象を絞る**（1テーブルだけ）
- **参照するシートを明示する**
- **配置先のパスを明示する**（Angular + NestJS のモノレポ構成）
- **追加しないよう明示する**（ハルシネーション防止）

### より良い例

```
specs/融資管理システム仕様書.xlsx の「テーブル定義」シートにある
loan_applications テーブルの定義を読み込んで、
Prisma スキーマの model を生成してください。
配置先は backend/prisma/schema.prisma です。

要件:
- カラムは仕様書の26カラムを全て含めること（過不足なく）
- 外部キー制約はリレーションとして定義
- インデックスは @@index で定義
- NOT NULL の場合は Optional にしない
- デフォルト値がある場合は @default() で指定

生成後、仕様書のカラム一覧と突き合わせて差異がないか自己チェックしてください。
```

ポイント:
- 具体的なカラム数を明記（検証基準になる）
- 技術的な要件を列挙
- **自己チェックを指示**（AIに自分の出力を検証させる）

### Angular コンポーネント生成のプロンプト例

```
specs/融資管理システム仕様書.xlsx の「画面仕様」シートにある
融資申請一覧画面（No.3）を Angular standalone component として実装してください。
配置先は frontend/src/app/features/loan-applications/loan-application-list/ です。

要件:
- standalone: true で作成すること
- テンプレートは別ファイル（.component.html）に分離すること
- スタイルは Tailwind CSS のユーティリティクラスを使用
- データ取得は HttpClient を使って backend の API を呼び出すこと
- 一覧表示・フィルタ・ソート・ページネーションを実装すること
- 表示項目は仕様書の「主要表示項目」に記載のもののみとすること

生成後、仕様書の表示項目一覧と突き合わせて過不足がないか自己チェックしてください。
```

ポイント:
- **Angular 固有の構成を明示する**（standalone, テンプレート分離）
- **配置先パスを Angular の慣習に合わせる**（feature ディレクトリ構成）
- **UI ライブラリを指定する**（Tailwind CSS）

## トラブルシューティング

### Excel ファイルが読めない場合

Claude Code はExcelファイルを直接読み取れます。ファイルパスが正しいことを確認してください。
相対パスで `specs/融資管理システム仕様書.xlsx` と指定するか、絶対パスを使用してください。

### 生成されたコードにエラーがある場合

```
このエラーを修正してください: [エラーメッセージを貼り付け]
```

エラーメッセージをそのまま貼り付けると、AIが適切に修正してくれます。

### Angular CLI / NestJS CLI のエラーが出る場合

```
Angular CLI と NestJS CLI がグローバルにインストールされているか確認してください:
npm list -g @angular/cli @nestjs/cli

インストールされていない場合:
npm install -g @angular/cli @nestjs/cli
```

### フロントエンドとバックエンドの接続がうまくいかない場合

```
Angular の開発サーバー（ng serve）から NestJS のバックエンドに
プロキシでリクエストを転送する設定（proxy.conf.json）を作成してください。
バックエンドは http://localhost:3000 で動作しています。
```

### 仕様書と異なるコードが生成された場合

```
生成されたコードを仕様書と突き合わせたところ、以下の差異がありました:
- [具体的な差異を記載]
仕様書の記載に合わせて修正してください。
```

具体的に差異を指摘することで、正確な修正が得られます。
