Claude Code Training

ハンズオン01:
Excel仕様書から
Claude Codeでアプリを生成する

融資管理システム（LoanFlow）を段階的に構築するハンズオン

目的・学べること

実際の業務で使われるExcel仕様書をClaude Codeに読み込ませ、
Webアプリケーションのコードを生成する一連の流れを体験します。

Excel読み取り能力

Claude CodeがExcelファイル(.xlsx)を読み取り、構造化されたデータを理解する能力を確認する

段階的なコード生成

仕様書からDBスキーマ → API → 画面を段階的に生成する進め方を習得する

ハルシネーション検出

AIが生成したコードの検証方法と、幻覚（hallucination）の検出方法を学ぶ

プロンプト設計

プロンプトの書き方による出力品質の違いを体感し、効果的なパターンを身につける

なぜExcel仕様書からの生成が重要か

  • 日本の企業ではExcelで管理された仕様書が依然として主流
  • 仕様書からコードへの変換は反復的で時間のかかる作業
  • 手作業での変換は転記ミスや解釈の齟齬が発生しやすい
  • Claude Codeは.xlsxファイルをネイティブに読み取り可能
  • AIによるコード生成で開発速度を大幅に向上できる
本ハンズオンの題材: 金融機関向け「融資管理システム（LoanFlow）」
融資申請 → 与信審査 → 承認 → 実行 → 返済管理 までの業務フローをカバーする仕様書（7シート構成）を使用

技術スタック

フロントエンド

  • Angular 17+（standalone components）
  • TypeScript 5.x
  • Tailwind CSS 3.x + Angular Material
  • Reactive Forms / HttpClient / Router

バックエンド

  • NestJS 10+ + TypeScript 5.x
  • Prisma 5.x（ORM）
  • PostgreSQL 16
  • Passport + JWT（認証）

ディレクトリ構成（モノレポ）

loanflow/ frontend/ # Angular src/app/ core/ # Guards, Services shared/ # 共有コンポーネント features/ # 機能モジュール models/ # 型定義 backend/ # NestJS src/modules/ # 機能モジュール prisma/ schema.prisma # DBスキーマ

テスト

  • Jest + Testing Library
  • Cypress（E2E）

前提条件・事前準備

必要な環境

  • Claude Code がインストール済み
  • Node.js 18以上
  • Angular CLI（npm i -g @angular/cli）
  • NestJS CLI（npm i -g @nestjs/cli）
  • PostgreSQL（ローカル or Docker）

事前準備コマンド

# ハンズオンディレクトリに移動 cd handson-01-excel-spec # （オプション）仕様書を再生成する場合 pip install openpyxl python generate_spec.py

Step 0

プロジェクト初期化

Claude Codeを起動し、モノレポプロジェクトの雛形を生成させます。

起動コマンド cd handson-01-excel-spec claude
プロンプト例 Angular 17+ (standalone components) のフロントエンドと NestJS のバックエンドで構成される モノレポプロジェクトを初期化してください。 - frontend/ ディレクトリに Angular CLI で新規プロジェクトを作成（Tailwind CSS を設定） - backend/ ディレクトリに NestJS CLI で新規プロジェクトを作成（Prisma をセットアップ） - パッケージマネージャーは npm を使ってください
ポイント: CLAUDE.md が自動的に読み込まれ、プロジェクトの技術スタック・ディレクトリ構成方針・コーディング規約がAIに伝わります。
Step 1

仕様書の読み込み

プロンプト例

specs/融資管理システム仕様書.xlsx を 読み込んで、全シートの内容を要約して ください。特に以下の点を確認したいです: - テーブル数とそれぞれのカラム数 - API エンドポイント数 - 画面数 - ビジネスルールの数
Tip: Claude Codeは.xlsxファイルをネイティブに読み取れます。パスの指定だけ注意。

検証チェックポイント

確認すべき数値
  • 全 7シート が認識されているか
  • テーブル数: 8テーブル
  • APIエンドポイント数: 27個
  • 画面数: 9画面
  • ビジネスルール数: 16個
  • 日本語カラム説明が正しく読めているか
Step 2

DBスキーマ生成

プロンプト例 specs/融資管理システム仕様書.xlsx の「テーブル定義」シートを読み込んで、 Prisma スキーマファイル（backend/prisma/schema.prisma）を生成してください。 以下の要件を守ってください: - テーブル定義シートの全テーブル・全カラムを漏れなく定義すること - 外部キー制約はPrismaのリレーションとして定義すること - インデックスは @@index で定義すること - データ型はPostgreSQL互換のPrisma型にマッピングすること - コメントを付与すること
検証チェックポイント
  • 全8テーブルが定義されているか
  • customers: 21カラム、loan_applications: 26カラム
  • 外部キーがリレーションとして定義されているか
  • NULL可否が Optional(?) で正しく表現されているか
  • 仕様書にないテーブルやカラムが追加されていないか

よくあるハルシネーション

  • users や branches テーブルを勝手に作成
  • audit_logs テーブルを追加（テーブル定義シートにない）
  • カラムの型制約を独自に追加・変更
Step 3

API実装

プロンプト例 specs/融資管理システム仕様書.xlsx の「API仕様」シートを読み込んで、 NestJS の Controller + Service として API を実装してください。 まず融資申請関連の API（No.3〜8）を実装してください: - GET /api/loan-applications（一覧取得・ページネーション・フィルタ対応） - GET /api/loan-applications/:id（詳細取得） - POST /api/loan-applications（新規作成） - PUT /api/loan-applications/:id（更新） - DELETE /api/loan-applications/:id（削除） - POST /api/loan-applications/:id/submit（申請提出） 以下の構成で実装してください: - backend/src/modules/loan-applications/ 配下にモジュール・コントローラー・サービスを作成 - リクエストのバリデーションは class-validator の DTO クラスで定義 - Prisma でのデータベース操作はサービス層に集約 - @UseGuards(JwtAuthGuard) による認証チェック
検証ポイント: APIパス・HTTPメソッド・リクエストパラメータ・ステータスコードが仕様書通りか。仕様書にないクエリパラメータやレスポンスフィールドが追加されていないかに注意。
Step 4

画面実装

プロンプト例 specs/融資管理システム仕様書.xlsx の「画面仕様」シートを読み込んで、 融資申請一覧画面（No.3）を Angular の standalone component として実装してください。 以下の要件を守ってください: - Tailwind CSS でスタイリングすること - 画面仕様シートの「主要表示項目」を全て表示すること - 「操作・アクション」を全て実装すること - フィルタ・ソート・ページネーションを実装すること - frontend/src/app/features/loan-applications/ 配下に配置すること - コンポーネントは standalone: true で作成すること
検証チェックポイント
  • URLが仕様書と一致（/loan-applications）
  • 表示項目: 申請番号 / 顧客名 / 商品名 / 金額 / ステータス / 申請日 / 担当者
  • 操作: 新規ボタン / 行クリック遷移 / CSV / フィルタ / ソート / ページサイズ
  • 仕様書にないUI要素が追加されていないか（例: 一括操作チェックボックスなど）
Step 5

ビジネスロジック実装

プロンプト例 specs/融資管理システム仕様書.xlsx の「ビジネスルール」シートを読み込んで、 以下のビジネスロジックを NestJS のサービスとして実装してください: 1. 与信スコア算出（BR-001〜BR-003, BR-016） 2. 融資限度額算出（BR-004） 3. 適用金利決定（BR-005） 4. 元利均等返済額計算（BR-006） 5. 承認フロー分岐（BR-007） 各ロジックは: - backend/src/modules/ 配下の適切なサービスに配置 - 計算ロジックは純粋関数としてメソッド内に実装（テストしやすく） - 計算式は仕様書の記載を正確に実装 - Jest によるユニットテストも同時に作成（*.spec.ts）
最重要: 与信スコアの閾値・融資限度額の倍率・金利計算の調整値・承認フロー分岐の金額閾値が仕様書と完全一致しているかを必ず確認。

ハルシネーション検出のコツ

AIは仕様書にない情報を「補完」する傾向があります。以下のポイントで検証してください。

1. 数値の正確性を最優先で確認

閾値・倍率・金利等は仕様書と1つずつ突き合わせる。
万円単位と円単位の変換ミスは特に頻出

// 仕様書: "300-500万" は 300万以上500万未満 if (income >= 3000000 && income < 5000000) // OK if (income < 5000000) // NG: 300万未満も含む
2. カラム数・API数を数える

仕様書の数と生成結果を比較。
多い → AIが勝手に追加
少ない → 見落とし

3. 「仕様書にないもの」を探す

AIが追加しがちなもの:

  • updatedAt / createdBy カラム
  • PATCHメソッドのAPI
  • 検索サジェスト機能
  • 追加のバリデーションルール
4. 境界値に注目する

「以上」「以下」「未満」「超」の区別。
金額の境界値（500万以下 vs 未満）。
0件・NULL・空文字のハンドリング。

ハルシネーション検出の実践テクニック

5. 仕様書を参照させながらセルフレビュー

Claude Code自身に生成コードと仕様書を突き合わせさせる

生成した与信スコア算出ロジック （backend/src/modules/credit/ credit-score.service.ts）を、 仕様書の「ビジネスルール」シート BR-001〜BR-003 と突き合わせて、 差異がないか確認してください。 差異がある場合は具体的に指摘して ください。

6. テストケースで検証する

手計算した期待値をテストケースとして追加するのが最も確実

以下のテストケースを追加して、 与信スコア算出が仕様書通りか 検証してください: - 年収400万・正社員・勤続5年 ・負債比率0.15・担保なし → 期待スコアを手計算して比較
鉄則: 数値を含むビジネスルールは必ず手計算した期待値でテストする

効果的なプロンプトのパターン

Bad - 曖昧すぎるプロンプト
仕様書を読んでアプリを作って

範囲が広すぎて、AIが独自の判断で多くの仕様を「補完」してしまう

Good - 対象を絞り、制約を明示
specs/融資管理システム仕様書.xlsx の 「テーブル定義」シートにある loan_applications テーブルの定義を 読み込んで、Prisma スキーマの model を 1つだけ生成してください。 配置先は backend/prisma/schema.prisma カラムは仕様書に記載されたものだけを 含め、追加しないでください。

良いプロンプトの4原則

  • 対象を絞る - 1テーブル、1画面、1APIグループずつ
  • 参照シートを明示する - 「テーブル定義」「API仕様」など具体的に
  • 配置先パスを明示する - ディレクトリ構成を指定
  • 追加禁止を明示する - 「仕様書に記載されたものだけ」でハルシネーション防止

より良いプロンプトのパターン

Best Practice specs/融資管理システム仕様書.xlsx の「テーブル定義」シートにある loan_applications テーブルの定義を読み込んで、 Prisma スキーマの model を生成してください。 配置先は backend/prisma/schema.prisma です。 要件: - カラムは仕様書の26カラムを全て含めること（過不足なく） - 外部キー制約はリレーションとして定義 - インデックスは @@index で定義 - NOT NULL の場合は Optional にしない - デフォルト値がある場合は @default() で指定 生成後、仕様書のカラム一覧と突き合わせて差異がないか自己チェックしてください。
具体的な数値を明記

「26カラム」のように数を指定 → 検証基準になる

技術要件を列挙

リレーション・インデックス・デフォルト値など具体的に

自己チェックを指示

AIに自分の出力を仕様書と突き合わせて検証させる

CLAUDE.md の活用

CLAUDE.md はClaude Codeが自動で読み込むプロジェクト設定ファイル。プロンプトに書かなくても前提条件がAIに伝わります。

記載すべき内容

# プロジェクト概要 金融機関向けの融資管理システム # 技術スタック - Angular 17+ (standalone components) - NestJS 10+ / Prisma 5.x - PostgreSQL 16 # ディレクトリ構成方針 frontend/ ... / backend/ ... # コーディング規約 - ファイル名はケバブケース - DBカラム名はスネークケース - 金額は整数（円単位）で扱う # AI への指示方針 - 仕様書に記載のないカラムや APIを勝手に追加しないこと

効果

  • 毎回のプロンプトで技術スタックを繰り返さなくてよい
  • コーディング規約が一貫して守られる
  • ディレクトリ構成がブレない
  • 「追加しないこと」の指示が常に有効

AI への指示方針（抜粋）

  • 仕様書に厳密に従うこと
  • 不明点は実装前に質問すること
  • テストコードも同時に生成すること
  • ビジネスルールの計算式は正確に実装すること

トラブルシューティング

Excelファイルが読めない

Claude CodeはExcelをネイティブに読み取れます。ファイルパスが正しいか確認してください。

# 相対パスまたは絶対パスで指定 specs/融資管理システム仕様書.xlsx
生成コードにエラーがある

エラーメッセージをそのまま貼り付けると適切に修正されます。

このエラーを修正してください: [エラーメッセージを貼り付け]
CLI のエラーが出る

Angular CLI / NestJS CLI がグローバルにインストールされているか確認。

npm list -g @angular/cli @nestjs/cli npm install -g @angular/cli @nestjs/cli
仕様書と異なるコードが生成された

差異を具体的に指摘することで正確な修正が得られます。

生成されたコードを仕様書と突き合わせた ところ、以下の差異がありました: - [具体的な差異を記載] 仕様書の記載に合わせて修正してください。

まとめ

段階的に生成する

DB → API → 画面 → ロジックの順に、小さな単位で生成・検証を繰り返す

ハルシネーションを意識する

数値の突き合わせ、カラム数の確認、「仕様書にないもの」の探索を習慣に

対象を絞る、シート名を明示、配置先を指定、追加禁止を明記、自己チェック指示

Key Takeaways

  • Claude CodeはExcel仕様書を読み取り、実用的なコードを生成できる
  • CLAUDE.md でプロジェクト設定を事前に定義すると品質が安定する
  • AIの出力は必ず仕様書と突き合わせて検証する（特に数値）
  • プロンプトの具体性と制約が出力品質を左右する
  • AIに自己チェックさせることでハルシネーションを軽減できる
