CHAPTER 00
このガイドの使い方
Claude Code を手元で動かしながら、Agent Team の全体像を学びます。

🎯 学習目標

このハンズオンを終えると、以下ができるようになります：

  • Claude Code でサブエージェントを定義・呼び出しできる
  • 専門性の異なる複数エージェントを設計できる
  • 並列実行で作業を高速化できる
  • SKILL.md を使って Agent Team パターンを自動化できる
  • Sub-agents と Agent Teams の違いを説明・使い分けできる
  • Agent Teams（実験的機能）でチームメンバー間 P2P 通信を体験できる

ツールバージョン確認コマンド

Anthropic API キー—echo $ANTHROPIC_API_KEY
Node.js（任意）v18 以上node --version
💡
API キーの設定 export ANTHROPIC_API_KEY="sk-ant-..." を .zshrc / .bashrc に追加しておくと便利です。

CHAPTER 01
Agent Team とは何か
マルチエージェントの概念とアーキテクチャを理解します。

🤔 なぜマルチエージェントが必要か

Claude Code（シングルエージェント）には限界があります：

問題詳細
コンテキスト上限1回の会話で扱える情報量に上限がある。大きなコードベースは全部見られない
処理速度複数の作業を順番にこなすため、時間がかかる
集中力の分散セキュリティも品質も同時に考えると、どちらも浅くなる

解決策は「専門家チームに分けて並列で動かす」こと、それが Agent Team です。

🏗️ アーキテクチャ — 3層構造

🎯 オーケストレーター（あなたが対話するClaude）
タスクを分解・委任・結果を統合する司令塔
↓ 指示・委任
🔒 セキュリティ担当
security-reviewer
📐 品質担当
code-quality-reviewer
📝 ドキュメント担当
doc-writer
↓ 結果のみを返す
🛠️ ツール — Read / Grep / Bash / Write / ブラウザ / MCP
ℹ️
重要なポイント 各サブエージェントは独自のコンテキストウィンドウを持ちます。互いの作業内容を知らないため、不要な情報が混入せず深い分析ができます。

📊 3大メリット

並列化 → 速度向上

独立したタスクを同時実行。直列の 1/N 倍 の時間で完了。
例：3エージェント並列 → 3倍速

コンテキスト分離 → 精度向上

各エージェントが専門領域だけに集中。不要な情報が混入せず、深い分析ができる。

権限の分離 → 安全性向上

エージェントごとに使えるツールを制限。読み取り専用エージェントと書き込みOKエージェントを使い分けられる。

🔍 Sub-agents vs Agent Teams — 何が違うのか

Claude Code には2種類のマルチエージェントがあります。このガイドの Chapter 02〜05 は Sub-agents、Chapter 06 は Agent Teams（実験的機能）を扱います。

Sub-agents（Chapter 02〜05）Agent Teams ⚗ 実験的（Chapter 06）
コンテキスト 独自のウィンドウ。結果はリーダーに返される 独自のウィンドウ。完全に独立
通信 リーダーにのみ結果を報告（一方向） チームメンバー間で直接メッセージ送受信（P2P）
調整 リーダーがすべての作業を管理 共有タスクリストで自己調整
最適な用途 結果のみが重要な焦点を絞ったタスク 議論・協力・相互検証が必要な複雑な作業
トークンコスト 低い：結果が要約されてリーダーに返る 高い：各メンバーが独立した Claude インスタンス
有効化 デフォルトで使える 環境変数 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 が必要
ℹ️
使い分けの基準 ワーカーが互いに通信する必要がない → Sub-agents
チームメンバーが調査結果を共有・相互検証・直接対話する必要がある → Agent Teams
CHAPTER 02
エージェントを定義する
Markdown ファイル 1 枚でサブエージェントを作る方法を学びます。

📄 エージェント定義ファイルの構造

サブエージェントは .claude/agents/ に置いた Markdown ファイルで定義します。

bash 
プロジェクト/
└── .claude/
    └── agents/
        ├── security-reviewer.md   ← エージェント定義
        ├── code-quality-reviewer.md
        └── doc-writer.md
markdown 基本テンプレート 
---
name: エージェント名（英小文字・ハイフン）
description: |
  いつこのエージェントを使うか。
  具体的なトリガーフレーズも書く。
model: claude-haiku-4-5-20251001
tools:
  - Read
  - Grep
---

ここにシステムプロンプトを書く。
エージェントの専門性・出力形式・注意点など。

🔑 description が最重要

オーケストレーターは description を読んでどのエージェントを使うか自動で判断します。

description: コードを確認します。

→「確認」が何を意味するか不明。いつ使えばいいか曖昧。

description: |
  コードのセキュリティレビューを行います。
  「セキュリティを確認して」「脆弱性を調べて」と
  言われたとき、またはPRレビューの際に使います。
  SQLインジェクション・XSS・認証バイパスを重点的に確認し、
  ファイル名・行番号・修正案を含む報告書を作成します。

→ いつ・何を・どうするかが明確。オーケストレーターが迷わない。

⚙️ モデルの選び方

エージェントごとに最適なモデルを選ぶことでコストと速度を最適化できます。

Opus 4.7
オーケストレーター向け
最高精度。複雑な判断・設計に最適
Sonnet 4.6 ⭐
汎用サブエージェント向け
バランス良好。コード生成・実装に
Haiku 4.5
軽量タスク向け
高速・低コスト。検索・確認作業に
💡
コスト削減のヒント 検索・確認・定型処理は Haiku に委ねるだけでコストを大幅削減できます。複雑な判断が必要なエージェントだけ Sonnet/Opus を使いましょう。

CHAPTER 03
専門エージェントを設計する
subagent_type の種類と、良いエージェント設計のパターンを学びます。

🤖 ビルトインエージェントタイプ

Claude Code には汎用の専門エージェントが組み込まれています：

タイプ得意なこと使うべき場面
Explore コードベースの高速探索 「〇〇が定義されているファイルを探して」「〇〇を参照している箇所は？」
Plan 実装計画の設計 「この機能をどう実装すればいい？」「アーキテクチャを考えて」
claude 汎用（デフォルト） 上記に当てはまらないすべてのタスク

さらに、.claude/agents/ に自分でエージェントを追加することで専門性を拡張できます。

🎯 良いエージェント設計の原則

単一責任 — 1エージェント = 1つの専門領域

「セキュリティも品質もドキュメントも見る」エージェントは作らない。
役割を絞ることでプロンプトが短くなり、精度が上がる。

✅ security-reviewer ✅ code-quality-reviewer ❌ all-in-one-reviewer

ツールは必要最小限に絞る

読み取りだけのエージェントに Write を与えない。
権限を絞ることで誤操作のリスクが下がり、エージェントも迷わない。

# 読み取り専用エージェント
tools:
  - Read
  - Grep

# 書き込みが必要なエージェント
tools:
  - Read
  - Grep
  - Write
  - Edit

出力形式を明示する

「どんな形式で返してほしいか」をシステムプロンプトに書く。
オーケストレーターが複数エージェントの結果を統合しやすくなる。

EXERCISE 3-1 新しい専門エージェントを作る

パフォーマンス分析専門のエージェントを作ってみましょう。

bash 
# ファイルを作成
touch .claude/agents/performance-analyzer.md
markdown .claude/agents/performance-analyzer.md 
---
name: performance-analyzer
description: |
  コードのパフォーマンス問題を分析します。
  「パフォーマンスを確認して」「速度改善の提案をして」「N+1問題を探して」
  と言われたとき、またはパフォーマンスチューニングの際に使います。
  計算量・データベースクエリ・メモリ使用量を確認し、改善案を報告します。
model: claude-haiku-4-5-20251001
tools:
  - Read
  - Grep
---

あなたはパフォーマンスエンジニアです。
依頼されたコードを以下の観点で分析し、改善提案をしてください。

## 確認観点

1. **N+1クエリ問題** — ループ内でクエリが実行されていないか
2. **計算量** — O(n²) 以上のアルゴリズムが使われていないか
3. **不要な処理の重複** — 同じ計算が何度も実行されていないか
4. **メモリ効率** — 大量データをメモリに全件読み込んでいないか
5. **インデックス不足** — 頻繁に検索するカラムにインデックスがないか

## 報告形式

問題ごとに以下を明記してください：
- 場所（ファイル:行番号）
- 現在の計算量（例: O(n²)）
- 改善後の計算量（例: O(n)）
- 具体的な修正コード
  • ファイルを保存する
  • Claude Code で「パフォーマンスを確認して」と入力し、新しいエージェントが選ばれることを確認する
  • description を変えてみて、呼ばれるトリガーが変わることを試す
CHAPTER 04
並列エージェント実行
複数のエージェントを同時に起動して、作業を大幅に高速化します。

⚡ 並列実行 vs 直列実行

直列実行（デフォルト）並列実行
実行順序A → B → C（順番に）A + B + C（同時に）
所要時間各タスクの合計時間最も時間のかかるタスクの時間
3タスク例30秒 × 3 = 90秒最長30秒 = 30秒
適した場面タスクが互いに依存する場合タスクが独立している場合

✍️ 並列実行を指示する書き方

「同時に」「並列で」「一度に」というキーワードで並列実行を促せます。

user-auth.js をセキュリティレビューして、
次に品質レビューして、
最後にドキュメントを確認して。
user-auth.js を以下の3つを同時に並列で実行してください：
1. security-reviewer でセキュリティ確認
2. code-quality-reviewer でコード品質確認
3. doc-writer でドキュメント不足の確認

全て完了したら結果をまとめて報告してください。

CHAPTER 05
SKILL.md で自動化する
Agent Team パターンを SKILL.md に定義して、ワンコマンドで呼び出せるようにします。

📋 SKILL.md とは

Claude Code の「カスタムスラッシュコマンド」です。.claude/commands/ に置いた Markdown ファイルが /ファイル名 コマンドとして使えます。

bash 
プロジェクト/
└── .claude/
    ├── agents/          ← エージェント定義
    └── commands/
        └── review.md    ← /review コマンドになる

🔧 並列レビュースキルを作る

bash 
mkdir -p .claude/commands
touch .claude/commands/review.md
markdown .claude/commands/review.md 
# /review — 並列コードレビュー

引数で渡されたファイルを3エージェントで並列レビューします。
引数がない場合は、直近で変更されたファイルを対象にします。

## 実行手順

1. 対象ファイルを特定する（引数 $ARGUMENTS またはgit diffで確認）
2. 以下を**同時に並列実行**する：
   - security-reviewer エージェントでセキュリティ確認
   - code-quality-reviewer エージェントでコード品質確認
   - doc-writer エージェントでドキュメント不足確認
3. 全エージェントの結果が揃ったら統合レポートを出力する

## 出力形式

# コードレビューレポート: {対象ファイル}
実行日時: {現在日時}

## 🔒 セキュリティ
{security-reviewerの結果}

## 📐 コード品質
{code-quality-reviewerの結果}

## 📝 ドキュメント
{doc-writerの結果}

## 🎯 優先修正リスト
{重大度・優先度順に並べた修正リスト}

🚀 スキルの呼び出し方

bash 
# 特定ファイルをレビュー
/review src/user-auth.js

# 引数なし → 直近の変更ファイルをレビュー
/review
💡
$ARGUMENTS 変数 スキル内で $ARGUMENTS を使うと、コマンド呼び出し時に渡した引数（/review src/auth.js なら src/auth.js）を参照できます。
EXERCISE 5-1 /review スキルを完成させる
  • .claude/commands/review.md を上記の内容で作成する
  • Claude Code を再起動（または /help でコマンド一覧を確認）
  • /review src/user-auth.js を実行する
  • スキルなしで個別に頼むのと比べて便利になったか確認する
EXERCISE 5-2（発展） 自分のユースケース用スキルを作る

以下のアイデアから1つ選んでスキルを作ってみましょう：

  • /standup — 昨日のコミットを分析してスタンドアップ報告文を生成
  • /pr-ready — PR前チェック（テスト通過・ドキュメント確認・セキュリティ確認）を並列実行

CHAPTER 06
Agent Teams ⚗ 実験的機能
実際に手を動かしながら、Agent Teams の全機能を体験します。3つのシナリオで段階的に学びます。

🔍 Sub-agents と何が違うのか

⚠️
実験的機能 — デフォルト無効 環境変数 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 で有効化が必要です。v2.1.32 以降。セッション再開・シャットダウン動作に既知の制限があります。

Sub-agents（Ch02〜05）との決定的な違いは通信方向です：

SUB-AGENTS（Ch02〜05）

🎯 オーケストレーター
↓ 指示（一方向）
🔒 security-reviewer
📐 quality-reviewer
↓ 結果のみ返す

AGENT TEAMS（Ch06）⚗

🎯 チームリーダー
↕ 双方向メッセージ
🔒 security
teammate-1
📐 quality
teammate-2
↔ P2P（互いに直接通信）
📋 共有タスクリスト
Sub-agentsAgent Teams ⚗
通信リーダーにのみ結果を報告（一方向）メンバー間で直接メッセージ送受信（P2P）
調整リーダーがすべて管理共有タスクリストで自己調整
最適用途結果だけが重要な集中タスク議論・相互検証・協力が必要な複雑作業
コスト低（結果が要約されて返る）高（各メンバーが独立した Claude インスタンス）

🚀 Scenario 1 — 最初のチームを起動する（15分）

3人のチームメンバーで src/user-auth.js を並列レビューし、P2P通信を体験します。

Claude Code のプロンプトに以下を入力してください：

claude-prompt
エージェントチームを作成して src/user-auth.js をレビューしてください。
以下の3つの役割でチームメンバーを生成してください（名前も指定します）：
- security という名前で：セキュリティの専門家として脆弱性を調査

各メンバーのセッションでメッセージのやり取りが起きているか確認してください。

💡
P2P通信が見えるタイミング security が脆弱性を発見すると、それを quality や advocate に共有します。advocate が反論を送ると security が再検討します。このやり取りが Sub-agents にはない Agent Teams の特徴です。

リーダーのプロンプトから特定メンバーへ追加指示を出せます：

claude-prompt
security チームメンバーに、db.js のコネクション管理も確認するよう伝えてください

または Shift + ↓ で security のセッションに直接切り替えて入力することもできます。

🚫
必ずリーダーからクリーンアップする チームメンバー側から指示するとリソースが不整合になります。
claude-prompt
チームメンバーを全員シャットダウンして、チームをクリーンアップしてください
  • 3つのチームメンバーが生成されたことを確認した
  • Shift + ↓ で各メンバーのセッションを見た
  • Ctrl + T で共有タスクリストを開いた
  • メンバー同士がメッセージをやり取りしているのを確認した
  • クリーンアップが完了した

📋 Scenario 2 — プラン承認ワークフロー（20分）

リスクの高い変更（リファクタリング・DB変更など）では、実装前にプランをレビューするフローが重要です。

チームメンバーに 実装前にプランを立てることを義務付け できます。

  1. チームメンバーが読み取り専用のプランモードで作業を開始する
  2. プランが完成したらリーダーに承認リクエストを送信する
  3. リーダーが承認 → 実装開始 / 却下（フィードバック付き）→ 再プランニング
claude-prompt
src/routes/users.js の N+1クエリ問題を修正するチームメンバーを1名生成してください。
名前は "refactor" にしてください。
変更前に必ずリファクタリングプランを立案し、私の承認を得てから実装してください。
テストカバレッジの言及があるプランのみ承認します。
👁
観察ポイント チームメンバーがプランモードで動作し、ファイルを読むだけで書き込まないことを確認してください。プランが完成するとリーダーに承認リクエストが届きます。

プラン承認リクエストが届いたら：

claude-prompt承認する場合
refactor のプランを承認してください。実装を開始してください。
claude-prompt却下してフィードバックを送る場合
refactor のプランを却下してください。
理由：テストの追加計画が含まれていません。
ユニットテストの追加も含めたプランを再提出してください。
👁
観察ポイント 承認後、チームメンバーがプランモードを抜けて実際にファイルを書き換え始めます。Shift + ↓ で refactor セッションに移動して、ファイル変更の様子を確認してください。
  • チームメンバーがプランモードで動いていることを確認した（ファイルを書き換えない）
  • プランが届いてリーダーが承認・却下を判断した
  • 却下した場合、メンバーが再プランニングしたことを確認した
  • 承認後に実際の実装が始まったことを確認した

🔬 Scenario 3 — 競合仮説デバッグ（30分）

これが Agent Teams の真価を最も体感できるシナリオです。複数のメンバーが異なる仮説で調査し、互いに反証し合いながら根本原因を絞り込みます。

💡
なぜこれが Sub-agents より優れるのか シングルエージェントや Sub-agents では、最初に見つかった「もっともらしい原因」にアンカリングしてしまいます。Agent Teams で複数のメンバーが独立して調査し、互いの理論を積極的に反証することで、生き残った仮説が真の根本原因である可能性が大幅に高まります。

src/db.js のコネクションリーク問題を3つの仮説で調査させます：

claude-prompt
src/db.js でコネクションリークが発生している原因を調査するエージェントチームを作成してください。
以下の3つの仮説を各チームメンバーに担当させてください：

- hypothesis-a：エラー時のコネクション未解放が原因
- hypothesis-b：競合状態（Race Condition）によるコネクション重複取得が原因
- hypothesis-c：コネクションタイムアウト未実装によるリソース枯渇が原因

各メンバーは自分の仮説を調査するだけでなく、
他のメンバーの仮説に対して積極的に反証してください。
科学的な議論のように、根拠のある反論を行ってください。
最終的に最も可能性の高い原因についてコンセンサスを取り、
修正提案をまとめてください。

Shift + ↓ で各メンバーのセッションを確認してください。

👁
以下を観察してください：
• hypothesis-a が「エラー時に releaseConnection が呼ばれていない」という証拠を見つける
• hypothesis-b が「pool.find は同期だが非同期呼び出しで競合が起きる」と指摘する
• hypothesis-c が「上記2つは正しいが、タイムアウトがないと永久にリークする」と追加
• メンバー同士が互いのメッセージに返答している様子

調査が停滞したり方向性がズレた場合、リーダーから修正指示を出せます：

claude-prompt
hypothesis-b チームメンバーに、非同期での競合状態の具体的なコードパスを
示すよう伝えてください
claude-prompt
全チームメンバーに現在の調査状況を共有し合うよう伝えてください
👁
期待される結果 db.js の問題は実際には3つの仮説すべてが当てはまります：
① catch 内で releaseConnection が呼ばれない
② pool.find が非同期競合に対応していない
③ タイムアウトがないため長期的にリークが蓄積する

各メンバーの仮説が互いの調査を深めることで、単一エージェントより包括的な診断ができます。
  • 3つの仮説メンバーが独立して db.js を調査した
  • メンバー間でメッセージのやり取りがあったことを確認した
  • 単一エージェントとの違いを体感できた
  • 最終レポートで複数の問題が統合されていることを確認した

🪝 Hooks で品質ゲートを実施する

Agent Teams 専用の Hooks を使うと、チームメンバーの動作に自動チェックを組み込めます。

フック名発火タイミング活用例
TeammateIdle チームメンバーがアイドル状態になろうとしているとき 終了コード 2 でフィードバックを送り、作業を継続させる
TaskCreated タスクが作成されているとき タスク名やスコープを検証し、不適切なタスクを防ぐ
TaskCompleted タスクが完了としてマークされているとき テスト通過・ドキュメント作成を確認してから完了を許可

チームメンバーが「もうやることがない」と判断してアイドルになるのを防ぐフックです：

json~/.claude/settings.json
{
  "hooks": {
    "TeammateIdle": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "echo '作業が残っていないか確認してください。タスクリストに未対応のタスクがあれば取得してください。'"
          }
        ]
      }
    ]
  }
}

タスク完了前にテストが通っているか確認するフックです：

json~/.claude/settings.json
{
  "hooks": {
    "TaskCompleted": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'echo タスク完了前確認: テストとドキュメントが含まれていますか？ >&2; exit 2'"
          }
        ]
      }
    ]
  }
}

終了コード 2 でフィードバックを送ると、タスクが完了としてマークされずチームメンバーが再確認します。

.claude/agents/ に定義した Sub-agents をチームメンバーとして呼び出せます。

💰 トークンコストとベストプラクティス

⚠️
コスト注意 Agent Teams は各チームメンバーが独立した Claude インスタンスです。3人のチームは同時に3倍のトークンを消費します。調整オーバーヘッドも加算されます。
チームメンバー数おすすめの用途
2〜3人レビュー・設計比較・単純な調査。バランスが良い
4〜5人複数の独立モジュールの並列実装・大規模調査
6人以上収穫逓減。調整コストが価値を超えることが多い

ベストプラクティス

  • 焦点を絞ったタスクから始める — コードレビュー・ライブラリ調査など、境界が明確でコードを書かないタスクで始める
  • タスクを適切にサイズ設定する — 1メンバーあたり5〜6タスクが目安。関数・テストファイル・レビューなど明確な成果物があるもの
  • ファイル競合を避ける — 2人が同じファイルを編集すると上書きが発生する。各メンバーに異なるファイルを担当させる
  • 生成プロンプトに十分なコンテキストを — メンバーはリーダーの会話履歴を引き継がないため、生成時に必要な情報を全て渡す
  • 定期的に監視・操舵する — 長時間放置するとコスト増・無駄な作業が積み重なる

🖥️ 分割ペインモード（tmux / iTerm2）

デフォルトの In-process モードでは全メンバーが同じターミナル内で動きます。
分割ペインモードでは各メンバーが独自のペインを持ち、全員の出力を同時に見られます。

bashtmux のインストール（macOS）
brew install tmux
bashtmux セッション内で Claude Code を起動
tmux new-session -s agent-team
claude
json~/.claude/settings.json
{
  "teammateMode": "tmux"
}
bashit2 CLI のインストール
npm install -g it2

その後 iTerm2 の Settings → General → Magic → Enable Python API を有効化してください。

ℹ️
自動検出 デフォルト設定 "auto" では、tmux セッション内なら分割ペイン、そうでなければ In-process が自動で使われます。

🔧 トラブルシューティング

  • In-process モードでは Shift + ↓ を押してチームメンバーをサイクルしてください（既に動いているが見えていない場合）
  • タスクが単純すぎてチームが不要と Claude が判断している場合は、より複雑なタスクを与えるか「チームを作成してください」と明示する
  • 分割ペインをリクエストした場合：which tmux で tmux がインストールされているか確認

チームメンバーの権限リクエストがリーダーにバブルアップして摩擦が生じます。
チームメンバー生成前に .claude/settings.json でよく使う操作を事前承認してください：

json.claude/settings.json
{
  "permissions": {
    "allow": ["Read(*)", "Grep(*)"]
  }
}

タスクが完了していないのにリーダーが「完了」と判断した場合：

claude-prompt
チームメンバーが完了するまで待ってから、作業を続けてください

チームのクリーンアップが不完全な場合、tmux セッションが残ることがあります：

bash
tmux ls                          # セッション一覧
tmux kill-session -t <name>     # 特定セッションを終了
tmux kill-server                 # 全セッションを終了（最終手段）

⚠️ 既知の制限事項

制限内容回避策
セッション再開なし /resume と /rewind は In-process チームメンバーを復元しない リーダーに新しいチームメンバーを生成するよう指示
タスクステータス遅延 チームメンバーがタスクを完了としてマークできず依存タスクがブロックされることがある リーダーにチームメンバーをナッジするよう指示
シャットダウン遅延 現在のリクエスト・ツール呼び出しを完了してからシャットダウン 少し待つ
1チームのみ リーダーは同時に1チームのみ管理できる 新チームを始める前に現在のチームをクリーンアップ
ネストなし チームメンバーは独自のチームを生成できない リーダーのみがチームを管理できる
分割ペイン制限 VS Code 統合ターミナル・Windows Terminal・Ghostty では分割ペイン不可 In-process モードを使用（デフォルト）
