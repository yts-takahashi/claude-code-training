CHAPTER 00
環境準備とサンプルプロジェクト紹介
Agent Teams を動かすための環境を整え、ハンズオンで使うサンプルプロジェクトを把握します。

🎯 このハンズオンで学ぶこと

Agent Teams は Claude Code の実験的機能です。複数の Claude Code インスタンスがチームとして協力し、P2P メッセージング・共有タスクリストを使いながら問題を解決します。このハンズオンを完了すると：

  • Agent Teams の概念と Sub-agents との違いを説明できる
  • チームを起動・操作・終了できる
  • チームメンバー間の P2P 通信を設計できる
  • プラン承認ワークフローでリスクを制御できる
  • Hooks でチームの品質ゲートを実装できる
  • 実務ユースケースにAgent Teams を適用できる

⚙️ 必要な環境

Claude Code のバージョン確認

Agent Teams は v2.1.32 以降で利用可能です。

bash
claude --version
# 2.1.32 以上であることを確認

古い場合はアップデート：

bash
npm update -g @anthropic-ai/claude-code

環境変数の設定

Agent Teams は環境変数で有効化します。

bash
# セッション内で一時的に有効化
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 設定の確認
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS
⚠️
永続化する場合 ~/.zshrc や ~/.bashrc に追加してください。ただし実験的機能であるため、本番環境への適用は慎重に判断してください。

サンプルプロジェクトへ移動

bash
cd handson-08-agent-teams/sample-project
ls src/

各モジュールの役割と含まれる問題

ファイル セキュリティ パフォーマンス 設計

💡

CHAPTER 01
Agent Teams とは
Sub-agents との違いを理解し、Agent Teams が解決する問題を把握します。

🤖 Sub-agents vs Agent Teams

Claude Code には「複数エージェント」機能が 2 種類あります。混同しやすいので、まず違いを整理します。

Sub-agents（安定版） Agent Teams（実験的）
起動方法 リーダーが Agent ツールで呼ぶ TeamCreate でチームを作成し、Agent(team_name, name) でメンバーをスポーン
通信モデル 一方向（リーダー→サブ、結果のみ返却） P2P双方向（誰でも誰にでもメッセージ送信可）
タスク管理 各エージェントが独立管理 TaskCreate / TaskUpdate でメンバーが共有タスクを管理
並行実行 コードで並列呼び出し可能 メンバーが同時に作業
エージェント定義 .claude/agents/*.md を自動参照 同ファイルをチームタイプとして利用可能
プラン承認 なし チームメンバーは読み取り専用プランモードで開始、リーダーが承認
終了担当 自動 リーダーのみが全員を終了できる

🎯 どちらを使うべきか？

ユースケース 推奨 理由
コードレビューを3つの観点で並列実行 Sub-agents シンプル・安定・結果を集約しやすい
セキュリティ診断 ↔ パフォーマンス診断の相互補完 Agent Teams 一方の発見をもう一方が深掘りできる
長時間の調査作業（途中で方針転換あり） Agent Teams P2P通信で動的に協議・再分担できる
リスクの高い変更（コード書き換えを含む） Agent Teams プラン承認ワークフローでリスク制御
単一の専門タスク（翻訳・フォーマット等） Sub-agents Agent Teams のオーバーヘッドが不要

⚠️ 既知の制限事項

⚠️
本番利用前に必ず確認してください
  • セッション再開（/resume）は未対応。中断したら最初から
  • タスクステータス更新に数秒の遅延がある場合がある
  • ネストチーム（チームの中のチーム）は非対応
  • チームのシャットダウンはリーダーが各メンバーに shutdown_request を送信して行う
  • アイドル通知（idle_notification）は自動配信される。返信は不要
CHAPTER 02
最初のチームを起動する
実際に Agent Teams を起動し、チームメンバーを追加してタスクを割り当てます。

🚀 チームの起動方法

Agent Teams は プログラマティック に構成します。ユーザーは Claude Code（リーダー）に自然言語で指示するだけで、Claude がツールを使ってチームを自動構成します。特別なキー操作は不要です。

環境変数を設定して Claude Code を起動

bash
cd handson-08-agent-teams/sample-project
CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 claude

起動後はリーダーの Claude Code に自然言語で指示するだけです。チームの構成は Claude が自動的に行います。

🔧 チーム構成の内部フロー

リーダーへ「チームを作って調査して」と指示すると、Claude は以下のツールを順に呼び出してチームを構成します：

1️⃣ TeamCreate(team_name) — チームを作成
↓
2️⃣ Agent(subagent_type,
team_name, "sec")
2️⃣ Agent(subagent_type,
team_name, "perf")
※ 複数メンバーは並列でスポーン可能
↓
3️⃣ TaskCreate(subject, description) — タスクを作成
↓
4️⃣ SendMessage(to: "メンバー名", message: "指示") — タスクを割り当て
↓
5️⃣ メンバーが作業完了 → teammate-message で自動報告
↓
6️⃣ SendMessage({type: "shutdown_request"}) — シャットダウン要求
↓
7️⃣ teammate_terminated が通知されて完了
ℹ️
エージェント定義の活用 .claude/agents/*.md に定義したエージェント（security-reviewer, performance-analyst）は subagent_type として指定できます。定義のシステムプロンプトがそのままメンバーの専門性になります。

📋 タスクとメッセージングの仕組み

チームメンバーは TaskCreate / TaskUpdate でタスクを管理し、SendMessage でリーダーや他のメンバーと通信します。

メッセージフロー例（リーダー視点）
// リーダーがセキュリティメンバーにタスクを割り当て
SendMessage(to: "sec", message: "src/users.js のSQLiを調査してください")

// メンバーが自動的に報告を返す（teammate-message として配信）
// { from: "sec", content: "3件のSQLiを発見: line 45, 78, 112..." }

// メンバーが完了後アイドル状態になる（idle_notification が届く）
// { type: "idle_notification", member: "sec" } — 返信不要

// 全作業完了後、シャットダウン
SendMessage(to: "sec",  message: {type: "shutdown_request"})
SendMessage(to: "perf", message: {type: "shutdown_request"})
// → teammate_terminated が通知されて完了

CHAPTER 03
チームを操作する
タスクの割り当て・メンバーへの指示・アイドル通知の扱い・シャットダウンをプログラマティックに操作します。

📋 タスクの割り当てと追跡

Agent Teams の操作はすべて プログラマティックです。リーダーがツールを呼ぶか、リーダーへの自然言語指示を通じてチームを制御します。

操作 ツール / 方法
タスク作成 TaskCreate(subject, description)
タスク割り当て SendMessage(to: "メンバー名", message: "指示内容")
完了報告受信 メンバーから teammate-message が自動配信される
タスク状態更新 メンバーが TaskUpdate でステータスを変更
追加指示 アイドル状態のメンバーに再度 SendMessage
シャットダウン SendMessage({type: "shutdown_request"}) → teammate_terminated

💤 メンバーのアイドル状態について

メンバーはターン終了後に自動的にアイドル状態になります。

アイドル通知の例（JSON形式で自動配信）
{
  "type": "idle_notification",
  "member": "sec",
  "message": "作業が完了し、次の指示を待っています"
}
ℹ️
アイドル状態の特徴
  • アイドル ≠ 終了。次の指示を待っている状態
  • SendMessage を送ると再び作業を開始する
  • アイドル通知への返信は不要（追加タスクがある場合のみ SendMessage）
  • 全員がアイドルになったらシャットダウンのタイミング

performance-analyst が誤ったファイルを調査している場合

メンバーが間違った方向に進んでいると判断したら、リーダーが直接 SendMessage で軌道修正します。

prompt
performance-analyst に SendMessage で
「src/search.js のパフォーマンス問題を優先して調査してください」
と指示してください。

🔍 タスクの状態を把握する

各メンバーが TaskCreate / TaskUpdate でタスクを更新すると、リーダーは TaskList で一覧を確認できます。

タスク状態の例

CHAPTER 04
P2P通信で協調させる
チームメンバーが互いにメッセージを送り合い、発見を組み合わせて深い分析を行います。

📡 P2P メッセージングとは

Agent Teams の最大の特徴が P2P（Peer-to-Peer）メッセージングです。メンバーはリーダーを経由せず、直接他のメンバーにメッセージを送ることができます。

Sub-agents: リーダー経由（通知のみ）
Sec Agent
→ 結果のみ →
リーダー
→ 転送 →
Perf Agent
Agent Teams: 直接 P2P 通信
Sec Agent
⟷ 直接通信 ⟷
Perf Agent

🔗 P2P通信が生む相乗効果

TaskTracker API の例では、セキュリティとパフォーマンスの問題が同じ箇所に重なっていることがあります。P2P 通信でこれを互いに補完できます。

search.js での相乗効果シナリオ

  1. security-reviewer が advancedSearch() で SQLインジェクションを発見
  2. security-reviewer が performance-analyst に直接メッセージ：「advancedSearch は SQLi があるが、WHERE句のパラメータ化でLIKE検索のインデックスも使えるようになるはず。その観点でも確認してほしい」
  3. performance-analyst が応答：「確認した。現在 title LIKE '%{q}%' でインデックスが使えない。プリペアドステートメント化 + full-text index の追加が最適解」
  4. 両エージェントが共有タスクリストに連携修正タスクを追加

💬 P2P通信を促すプロンプトの書き方

リーダーへの指示で、エージェント間の通信を明示的に促せます：

prompt — P2P通信を促す指示
security-reviewer が SQLインジェクションを発見したら、
その箇所を performance-analyst に共有してください。
performance-analyst はその情報をもとに、
パラメータ化クエリ化した場合のインデックス効率も
合わせて評価してください。

また、N+1クエリを performance-analyst が発見した場合、
security-reviewer にも通知して、一括フェッチ時の
認可チェックが適切かどうかを確認してもらってください。
💡
P2P通信のデザインポイント 「A が X を発見したら B に伝える」というトリガー→アクション形式で書くと、エージェントが自律的にコミュニケーションを取るようになります。

CHAPTER 05
プラン承認ワークフロー
チームメンバーは読み取り専用のプランモードで開始します。リーダーが承認するまでコード変更が起きません。

🔒 プランモードとは

Agent Teams では、チームメンバーは最初読み取り専用のプランモードで動作します。ファイルを読み、調査し、修正計画を立てますが、実際にファイルを変更したり、コマンドを実行したりはできません。

1️⃣ メンバーが調査 → プランを提出（読み取りのみ）
↓
2️⃣ リーダーがプランを確認・承認
↓
3️⃣ メンバーが実際のコード変更を実行
💡
なぜプラン承認が重要か リスクの高いコード変更（DBマイグレーション、認証ロジックの変更等）を複数エージェントが同時に行うと、予期しない競合やデータ破壊が起きる可能性があります。プラン承認ワークフローにより、「何を変更するか」を人間が確認してから「実際に変更する」というガードレールが設けられます。

📝 プランの構成要素

メンバーがリーダーに提出するプランには以下が含まれます：

プラン例（security-reviewer が提出）
## security-reviewer の修正プラン

### 対象: src/users.js

問題1 [高]: SQLインジェクション (line 45, 78, 112)
  現状: `SELECT * FROM users WHERE username = '${username}'`
  修正: プリペアドステートメントに変更
  影響範囲: 認証ロジック全体（テスト必要）

問題2 [高]: MD5パスワードハッシュ (line 23)
  現状: `crypto.createHash('md5').update(password)`
  修正: bcrypt (cost factor 12) に変更
  影響範囲: 既存ユーザーのパスワードは再ハッシュ不可 → 要移行戦略

問題3 [中]: ハードコード管理者トークン (line 8)
  現状: `const ADMIN_TOKEN = 'admin-secret-2024'`
  修正: 環境変数 ADMIN_TOKEN に変更
  影響範囲: デプロイ設定の変更が必要

承認をお願いします。承認後、上記の順番で修正を実施します。

✅ リーダーの承認フロー

リーダーはプランを受け取り、以下のアクションを取れます：

アクション 効果
承認（「承認します」と返信） メンバーが実際のコード変更を開始
条件付き承認（「問題1,2のみ承認」） 指定した問題のみ修正を実施、残りは再提出
差し戻し（「再検討してください」） メンバーがプランを修正して再提出
中断（Escape） メンバーの作業を中断

CHAPTER 06
Hooks で品質ゲートを作る
Agent Teams 専用の Hook イベントを使って、チームの作業を自動監視・制御します。

🪝 Agent Teams の Hook イベント

Agent Teams には 3 つの専用 Hook イベントがあります（通常の Claude Code の Hook に加えて）：

イベント名 発火タイミング 主な用途
TeammateIdle チームメンバーが待機状態になった時 次のタスクを自動割り当て・進捗通知
TaskCreated 共有タスクリストに新タスクが追加された時 タスクの品質チェック・優先度設定・外部通知
TaskCompleted タスクが完了マークされた時 成果物の検証・次のタスクのトリガー・レポート
ℹ️
Hook の終了コード Hook スクリプトが exit 2 を返すと、Claude Code に「フィードバックあり」として stdout の内容が Claude に送られます。exit 0 は成功（フィードバックなし）です。

🔁 Hook のフロー図

TaskCompleted Hook の品質ゲート

→

→
合格: exit 0
TaskCompleted
→
品質チェックスクリプト
→
不合格: exit 2 + フィードバック
→

🛠 品質ゲートを実装する

TaskCompleted 品質ゲートの作成

セキュリティ問題の修正が完了したとき、修正漏れがないか自動チェックするスクリプトを作成します。

bash — .claude/hooks/task-quality-gate.sh
#!/bin/bash
# TaskCompleted Hook: セキュリティ修正の品質ゲート

TASK_DESCRIPTION="${CLAUDE_TASK_DESCRIPTION:-}"
MODIFIED_FILES="${CLAUDE_MODIFIED_FILES:-}"

# タスクがSQLインジェクション修正に関するものかチェック
if echo "$TASK_DESCRIPTION" | grep -qi "sql\|injection\|sqli"; then

  # 変更されたJSファイルを確認
  for file in $(echo "$MODIFIED_FILES" | tr ',' '\n' | grep '\.js$'); do
    if [ -f "$file" ]; then

      # テンプレートリテラルでSQLを組み立てているパターンを検索
      if grep -n "WHERE.*\${" "$file" | grep -v "^\s*//"; then
        echo "❌ 品質ゲート失敗: $file にまだSQLインジェクションの疑いがあります"
        echo "上記の行を確認し、プリペアドステートメントを使用してください"
        exit 2  # フィードバックをClaudeに送る
      fi

    fi
  done

  echo "✅ SQLインジェクション品質ゲート通過"
fi

exit 0
bash — .claude/hooks/notify-idle.sh
#!/bin/bash
# TeammateIdle Hook: メンバーがアイドルになったら次のタスクを提案

TEAMMATE_NAME="${CLAUDE_TEAMMATE_NAME:-teammate}"
PENDING_TASKS=$(cat .claude/pending-tasks.txt 2>/dev/null | head -1)

if [ -n "$PENDING_TASKS" ]; then
  echo "$TEAMMATE_NAME が待機中です。次のタスク候補: $PENDING_TASKS"
  exit 2  # リーダーにタスク割り当てを促す
fi

exit 0

ユースケース集と実践課題
Agent Teams が真価を発揮するユースケースを把握し、総まとめの実践課題に挑戦します。

🗂 Agent Teams が特に有効なユースケース

🔍 マルチ視点コードレビュー

同一コードに対して「セキュリティ」「パフォーマンス」「可読性」の専門家が同時にレビューし、互いの発見を補完します。

prompt例
PR #123 を3人でレビューしてください。
security-reviewer: 脆弱性・認証・認可
performance-analyst: クエリ・キャッシュ・N+1
3人目のメンバーを追加して: 設計・可読性・テストカバレッジ
重複した問題は互いに確認し合い、最終的に統合レポートを作成してください。

🐛 仮説対決デバッグ

本番障害の原因を複数エージェントが異なる仮説で調査し、互いに反証作業を行います。生き残った仮説が真の根本原因です。

prompt例
本番で毎晩2時に APIが504エラーになります。
Agent1（仮説A）: 「夜間バッチのDB競合が原因」という前提で調査
Agent2（仮説B）: 「メモリリークで2時間後にOOMが起きている」という前提で調査
それぞれ自分の仮説を証明しつつ、相手の仮説の証拠も探してください。

📊 大規模リファクタリングの分担

複数モジュールにまたがるリファクタリングを、モジュールごとに分担して並列実行します。

prompt例
コールバックをすべてasync/awaitに変換してください。
Agent1: src/users.js, src/auth.js を担当
Agent2: src/issues.js, src/comments.js を担当
変換中に型エラーを発見したら互いに共有し、
変換後は互いのコードをクロスレビューしてください。

📝 仕様書からの実装

設計者とコーダーとテスター役のエージェントがリアルタイムで連携して実装を進めます。

prompt例
検索機能を実装してください（仕様: search.md）。
Agent1（設計者）: APIのインターフェースとDBスキーマを設計し、Agent2に共有
Agent2（実装者）: 設計を受けて実装し、不明点はAgent1に確認
リーダー（私）: 設計レビューと最終確認を担当

❌ Agent Teams が向かないケース

ケース 理由 代替手段
単純な1ファイル変更 チーム起動のオーバーヘッドが不要 通常の Claude Code
結果を集約するだけのパターン Sub-agents の方がシンプル・安定 Sub-agents + AgentToolの並列呼び出し
セッションを跨いで継続する作業 /resume 未対応 Sub-agents（セッション跨ぎ可能）
VS Code 内での利用 分割ペイン非対応（インプロセスのみ） ターミナルで tmux + Claude Code
